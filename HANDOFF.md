## Context Notes

- **`qb_host_query` is the foundation for edition-gated tools (#55, #66) AND the source of `maxQbxmlVersion` for any future SDK-version-gated feature.** Downstream tools that gate on edition should never parse `productName` — they should branch on `edition` / `isEnterprise` / `isAccountant`. Edition-to-feature mapping is the downstream tool's responsibility, not this tool's (the host tool reports what QB reports; downstream tools compose `edition` with their own subscription/feature checks).
- **Sim seed is fixed at `PremierAccountant`.** Tools that gate on `edition === "Enterprise"` will always fail in sim mode. That's expected — sim is for testing tool wiring, not for edition gating (which is a live-mode concern). If a future sim-mode test needs a different edition, the cleanest path is a `session.peekHostInfoCache()`-replacing test accessor; don't add a `setHostInfoForTesting` to production without a real need.
- **The cache lives at the session manager, not in the tool.** Two reasons: (1) keeps the gating signal one-call-away for any future tool that needs edition info internally (without going through the MCP boundary), and (2) survives across calls to the same session — the operator can call `qb_host_query` once per session and trust the result for the lifetime of the session.
- **`switchCompanyFile` clear of the cache is defensive, not load-bearing.** Under QBXMLRP2 localQBD the host (QB Desktop process) never changes across company-file switches. Clearing on switch is cheap insurance against a future remote-mode setup; the cost is one round trip on a rare event.
- **Live verification carry — the deferred SCF labels are still best-guess.** Both #54 (SCF) and #82 (Host) ship with sim-side complete and live-side verified-by-construction. The QBXML logger (#83) is the diagnostic primitive that turns the first live exercise into structured information rather than a debug session.
- **Carried gotchas (unchanged from prior handoffs):**
  - **Three lists must stay in sync** for transaction types: `buildDeleteRequest` in [src/qbxml/builder.ts](src/qbxml/builder.ts), inline `isTransaction` in `manager.deleteEntity` ([src/session/manager.ts](src/session/manager.ts)), and `isTransactionType` in [src/session/simulation-store.ts](src/session/simulation-store.ts). Canonical 16-type set: Invoice, Bill, Estimate, SalesReceipt, CreditMemo, PurchaseOrder, JournalEntry, Deposit, Transfer, Check, BillPaymentCheck, BillPaymentCreditCard, ReceivePayment, SalesOrder, CreditCardCharge, CreditCardCredit.
  - **AR-side `Customer.Balance` discount math is correct; AP-side is NOT.** `applyTxnApplications` / `reverseReceivePaymentApplication` (AR) move `Customer.Balance` by `appliedSum + discountSum`. The mirror functions `applyBillPayment` (AP) still move `Vendor.Balance` by `appliedSum` only. Future `qb_bill_write_off` will need the parallel fix.
  - Dispatch-order in `processRequest`: non-entity-typed `*QueryRq` / `*ModRq` branches (`CustomDetailReportQueryRq`, `GeneralDetailReportQueryRq`, `TransactionQueryRq`, `ClearedStatusModRq`) MUST precede the `endsWith("QueryRq")` / `endsWith("ModRq")` catch-alls. **Host is NOT in this category** — it intentionally goes through the catch-all `handleQuery` because the response shape is a singleton mapped from the Host store.
  - Synthetic statusCode sentinels: `9001` = read-only gate; `9002` = idempotency-key conflict; `9003+` reserved. All mapped in [src/util/qb-status-codes.ts](src/util/qb-status-codes.ts).
  - Iterator wire names diverge: `iterator`/`iteratorID` on request, `iteratorRemainingCount`/`iteratorID` on response.
  - fast-xml-parser does NOT decode numeric character entities — use `decodeXmlEntities` in [parser.ts](src/qbxml/parser.ts).
  - Live verification requires `C:\nvm4w\nodejs\node.exe` v20.20.2 (system PATH v22 breaks winax).
  - QBXMLRP2 cannot OPEN a `.qbw`, only attach to one QB Desktop has already loaded.
  - BillPayment* total is on `TotalAmount`, not `Amount` — coalesce `TotalAmount ?? Amount` for BillPaymentCheck / BillPaymentCreditCard; Check / CreditCard* / Transfer use `Amount` directly.
  - Bill ItemLineAdd in test fixtures should pass explicit `Amount` (or use Rate) — the sim's `convertLineAddToRet` computes `Amount = Quantity * Rate` and Bill ItemLineAdd's `Cost` field isn't picked up there.
