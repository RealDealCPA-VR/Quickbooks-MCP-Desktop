## Last Session Summary

- **Phase 18 #84 CLOSED — auto-reconnect on transient `QBXMLRP2` failures.** Retry loop landed inside [src/session/manager.ts](src/session/manager.ts) `sendRequest`'s live branch — single chokepoint covers every wire call without changing the typed mutation helpers (`addEntity` / `modifyEntity` / `deleteEntity` / `executeBatchAdd`). Three new layers:
  - **Pure classifier** `isTransientLiveError(err)` — conservative whitelist matching `0x80040408` HRESULT (hex), `-2147220472` (decimal form winax sometimes emits), and the descriptive `"QBSession not open"` text (case-insensitive). Deliberately does NOT match `0x80040409` / RPC_E_* / QB-status-code-shaped errors — those have no evidence of being recoverable by reconnect and would either waste the retry budget or silently mask real failures. Exported from `manager.ts` for direct unit testing without going through the manager.
  - **Frozen schedule** `RECONNECT_BACKOFF_MS = Object.freeze([250, 500, 1000])` — three retries after the initial attempt, total worst-case sleep budget 1.75s. Exported and pinned by test so future tuning is intentional.
  - **Retry loop** in new private `sendLiveRequestWithRetry`: per-attempt `logRequest` / `logResponse` / `logError` so QB_DEBUG_QBXML log files show the full retry sequence with separate seq markers; per-retry `console.error` for live operability without requiring debug logging. After each transient failure, `reconnectAfterTransientError` does best-effort `EndSession` + `CloseConnection` (swallowing throws — ticket likely already dead), nulls `rp` + `session`, and re-runs `openSession`. Reconnect-during-reconnect failure throws a wrapped error naming BOTH the reconnect failure and the original transient error (retry budget forfeit at that point — if QB Desktop won't accept a fresh `OpenConnection2` right now, looping won't help).
  - **State preservation across reconnect:** `readOnly` flag (intentionally), `idempotencyCache` (per-companyFile, only cleared by `switchCompanyFile`), `hostInfoCache` (installation-scoped, same QB process). The retry chokepoint sits BELOW the idempotency cache — a wire-level retry uses the same envelope on the same fingerprint, so QB-side behavior is identical to a single longer attempt. Putting retry above the typed helpers would have risked double-posting after a half-successful first attempt.
  - **Sleep impl is an instance field** `sleepImpl: (ms) => Promise<void>` — overridable via `(sm as any).sleepImpl = stub` matching the existing test-seam pattern (`store.processRequest` patches in qbxml-logger.test.ts).
  - **21 new tests** in [tests/connection-robustness.test.ts](tests/connection-robustness.test.ts) across 3 layers: (1) pure classifier — hex/decimal/descriptive matches, case-insensitivity, non-Error inputs (string, null, undefined, plain-object-with-message-not-an-Error), RPC_E_* deliberately excluded; (2) `RECONNECT_BACKOFF_MS` pinned `[250, 500, 1000]` + `Object.isFrozen`; (3) end-to-end via fake-live manager (patches `simulationMode=false`, stubs `rp.ProcessRequest` with a behavior queue, stubs `openSession` to re-install fakes without loading winax, captures `sleepImpl` durations) — initial success path (no retry, no sleep), single-transient-then-success (one 250ms sleep, one reconnect, two `ProcessRequest` calls), persistent 4-fail (sleeps 250 → 500 → 1000, four `ProcessRequest` calls, three reconnects, final error thrown), non-transient throws immediately, mixed transient-then-non-transient (second error wins with one sleep), reconnect-failure wraps both causes, per-retry `console.error` visibility, matching backoff per attempt, `readOnly` survives reconnect, descriptive-text-only match without hex code, swallows `EndSession`/`CloseConnection` throws during teardown, sim mode does NOT retry.

## Context Notes

- **#84 retry chokepoint sits INSIDE `sendRequest`, below the idempotency cache.** This is the load-bearing design choice — putting retry above the mutation helpers would risk double-posting after a half-successful first attempt (QB committed but response was lost). The wire-level retry reuses the same envelope on the same fingerprint, so QB sees one logical attempt regardless of how many physical retries fired. Future agents must NOT lift the retry up to `addEntity` / `executeBatchAdd` / etc.
- **#84 classifier whitelist is intentionally narrow.** Three patterns match: `0x80040408` hex, `-2147220472` decimal, `"QBSession not open"` text — all the same underlying signal. If a real-world transient surfaces with a different string format, extend the classifier + add the format to [tests/connection-robustness.test.ts](tests/connection-robustness.test.ts). Preference: "miss a real transient" (one-line fix) over "retry on a permanent error" (silent latency tax). Do NOT preemptively add `0x80040409` / RPC_E_* — no evidence they're recoverable by reconnect.
- **#84 retry budget is fixed at 4 attempts / 1.75s.** Tuning constant lives in `RECONNECT_BACKOFF_MS` in [src/session/manager.ts](src/session/manager.ts); changing it requires updating the test that pins the schedule. If a recovery pattern needs more time, add a fourth tier (e.g. 2000ms) rather than stretching existing values.
- **#84 fake-live test seam** (`(sm as any).simulationMode = false` + patched `rp` + patched `openSession` + `sleepImpl` override) is a load-bearing test pattern. Future renames of those private fields will break the retry tests; the test file documents this dependency.
- **State preservation across reconnect** is documented invariant: `readOnly`, `idempotencyCache`, `hostInfoCache` all survive. If a future field is added to `QBSessionManager` that has per-session semantics rather than per-process, `reconnectAfterTransientError` must explicitly clear it; the docstring calls out the existing exempt set.
