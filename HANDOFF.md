- **`qb_journal_entry_list` / `_create` / `_update` / `_delete`** in new module [src/tools/journal-entries.ts](src/tools/journal-entries.ts). Four tools. JE is the structural outlier of the transaction family — every other transaction posts from a single side (invoice → AR, bill → AP) but a JE is two-sided: every entry carries a `debits` array AND a `credits` array, each line naming a GL account. The tool API uses two separate top-level arrays (cleaner than mixing into one with a discriminator), with a small `buildJELineAdd` / `buildJELineMod` helper to project line objects to QBXML shape (`AccountRef`, `EntityRef`, `ClassRef`, `Memo`, `Amount`).
- **Hard balance invariant `sum(debits) === sum(credits)` enforced at the simulation layer with statusCode `3030`** ("request data is invalid"). Validate-first ordering at every persist boundary — both create and update return 3030 BEFORE persisting on any imbalance, so a doomed entry never lands in the store and a doomed mod leaves the existing entry untouched. The validator (`validateJournalEntryBalance` at [simulation-store.ts](src/session/simulation-store.ts)) reads `JournalDebitLineRet` / `JournalCreditLineRet` sums via a small `sumLineAmounts` helper and tolerates 0.005 of float drift.
- **Per-line `EntityRef` (Customer/Vendor/Employee/OtherName) is recorded but does NOT move that entity's balance** in this server's first cut. Real QB moves AR/AP per-line when a debit/credit on a Customer/Vendor line lands; that bookkeeping is meaningfully more involved than a single `adjustPartyBalanceForTxn` call (each line is its own posting, sign depends on debit-vs-credit + AR-vs-AP, single JE can touch many entities) and is **deferred**. Surfaced clearly in the JE tool descriptions and the README intro paragraph.

- **The JE balance invariant uses 0.005 tolerance** to absorb IEEE 754 float drift on cents-level sums (e.g. 0.1 + 0.2 !== 0.3 exactly). Real QB rejects to the cent; this server matches that user-facing precision while not blowing up on representable-but-imprecise sums.
- **`computeTotals` at [simulation-store.ts](src/session/simulation-store.ts)** now handles `Invoice | Estimate | SalesReceipt | Bill | CreditMemo | PurchaseOrder | JournalEntry`. The Subtotal-set list is still `Invoice | Estimate | SalesReceipt | CreditMemo` (unchanged) — neither PurchaseOrder nor JournalEntry sets Subtotal. The post-mod recompute conjunction in `handleMod` is now `Bill | Invoice | Estimate | SalesReceipt | CreditMemo | PurchaseOrder | JournalEntry` — only `Bill` needs the `delete updated.AmountDue` pre-clear.
- **The JE re-validation in handleMod is intentionally defensive** — it runs on header-only mods too (where `lineModResult.lineModKeys.size === 0`), even though the entry should already be balanced from the prior add/mod. Costs nothing if balanced and would catch a corrupted entry. The re-validation is OUTSIDE the `lineModKeys.size > 0 && ...` block (which controls the `computeTotals` re-run) — but inside `if (entityType === "JournalEntry")` and after the conjunction block, so for header-only mods it just re-derives totals from the existing line set and validates.
- **`adjustPartyBalanceForTxnMod` and `adjustPartyBalanceForTxn` are unchanged** — JE doesn't call either (per-line entity-balance moves are deferred). When that work is picked up, it'll need a new helper that walks debit/credit line arrays and routes per-line `EntityRef` + `Amount` through `adjustEntityBalance`, with sign matrix: AR debit lines increase customer Balance, AR credit lines decrease it; AP credit lines increase vendor Balance, AP debit lines decrease it; non-AR/AP accounts have no entity-balance side effect even if EntityRef present.
- **Verification gotcha (carried from prior handoffs)** — `handleQuery` filters require **uppercase** `TxnID` / `RefNumber` / `FullName` in the filter object when calling `session.queryEntity` directly. Tools translate from lowercase correctly. The verify-je.mjs script invoked tools through their captured handlers (cleaner — exercises the full tool-surface contract end-to-end), which avoids this trap entirely. Future verification scripts should prefer this pattern.
