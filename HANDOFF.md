## Context Notes

- **#57a carry-policy decision is intentional and differs from #convert.** `qb_estimate_convert_to_invoice` carries source's `RefNumber` by default ("the engagement's number is its identity"); `qb_invoice_duplicate` does NOT carry `RefNumber` ("a duplicate is a NEW invoice with a NEW number — carrying causes collisions in real QB"). Same composition pattern, opposite default. If the operator pushes back on this, the override path (`refNumber: <source.RefNumber>`) lets them opt in. Likewise `TxnDate` and `DueDate` are explicitly NOT carried — the duplicate use case is "next month's billing cycle," not "second copy of the same dated invoice." Carry list is unchanged from #convert: `ClassRef`, `TermsRef`, `SalesRepRef`, `PORefNumber` (header refs that ARE engagement-identity, not time-identity).
- **Composite tools are tool-layer-only.** No new wire types, no new `QBSessionManager` methods, no new `simulation-store` handlers — `qb_invoice_duplicate` is pure orchestration over existing `queryEntity` + `addEntity`/`addEntityIdempotent`. This is the cleanest pattern for "workflow stand-in" tools where QBXML has the primitives but no single RPC. Mirrors the `qb_estimate_convert_to_invoice` template. Apply the same shape to #57 (write-off = create credit memo + apply), bill/SR/JE duplicate variants, and any other operator-pain-driven composite.
- **`addEntityIdempotent` returns `{ entity, replayed }`.** The "no key" branch needs `{ entity: ..., replayed: false }` synthesis to keep the destructuring uniform — same shape as `qb_invoice_create` and `qb_estimate_convert_to_invoice`. Don't refactor — the duplication is intentional and keeps each tool readable in isolation.
- **InvoiceLineRet → InvoiceLineAdd mapping drops TxnLineID.** Real QB rejects an `InvoiceLineAdd` with a `TxnLineID` field ("can't specify a line ID on an Add"). New invoices generate their own line IDs. Same rule applies to bill/SR/JE/PO duplicate when those land.
- **Carried gotchas (unchanged from prior handoffs):** builder always wraps `addEntity` data with the `*Add` envelope (pass flat fields, not `{InvoiceAdd: {...}}`); iterator wire names diverge (`iterator`/`iteratorID` on request, `iteratorRemainingCount`/`iteratorID` on response); fast-xml-parser does NOT decode numeric character entities (use `decodeXmlEntities` in parser.ts); live verification requires `C:\nvm4w\nodejs\node.exe` v20.20.2 (system PATH v22 breaks winax); QBXMLRP2 cannot OPEN a `.qbw`, only attach to one QB Desktop already has loaded.
- **Dispatch-order subtlety in `processRequest`** ([src/session/simulation-store.ts](src/session/simulation-store.ts)). The `key === "ClearedStatusModRq"` branch MUST precede the `key.endsWith("ModRq")` catch-all. Any future non-entity-typed `*ModRq` must slot in similarly. Inline comment documents this — unchanged by #57a (which uses Query + Add only, no new Mod variants).
- **Synthetic statusCode sentinels:** `9001` = read-only gate; `9002` = idempotency-key conflict; `9003+` reserved. All mapped in [src/util/qb-status-codes.ts](src/util/qb-status-codes.ts) and surfaced through the existing tool error wrapper with `humanReadable`.
