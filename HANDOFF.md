## Context Notes

- **Phase 3 closure**: All transaction-completeness items (4–9) are now done. The system supports invoice CRUD, bill CRUD, payment receive + apply, bill pay (check + credit card), and the full `*Mod` line-mod flow on invoices and bills. Item 8's payment-apply Mod path is the only `*Mod` flow that currently exists for payment-class entities; bill payments don't have a Mod path (Phase 4 work if needed).
- **`applyBillPayment` is parallel to `applyReceivePayment`, NOT generic.** A future refactor could extract `applyTxnPayments(targetEntityType, balanceField, partyType, refField, hasAppliedAmount)` if a third payment kind lands. For now CLAUDE.md's "three similar lines is better than a premature abstraction" rule wins — there are exactly 2 call sites and the divergent fields (no AppliedAmount/UnusedPayment on bill payments, different store/balance/ref) make the abstraction shape uncertain. Don't extract until forced.
- **`Bill.IsPaid` is now set by `computeTotals` (Item 9 addition).** Existing seed bills are NOT pushed through `computeTotals` (they're inserted directly via `seedData()`), so they don't carry `IsPaid` unless modified. New bills via `qb_bill_create` always have it. `qb_bill_update` re-runs `computeTotals` and sets it. Filter `qb_bill_list { paidStatus: "PaidOnly" }` works on the new field but won't match old seed bills until they're modified — flag if it bites.
- **`qb_bill_payment_list` fan-out vs filter behavior.** The tool fans out across both BillPayment* stores via `Promise.all` when no `paymentType` is given. The simulation's `EntityFilter` (vendor name) IS honored on each per-store query. `MaxReturned` is applied PER-STORE on the fan-out path — the field's `.describe()` calls this out. If the operator passes `maxReturned: 10` and both stores have ≥10 matching records, they'll get up to 20 back. Document if this surprises a future caller.
- **Parser arrayElements check.** [src/qbxml/parser.ts:28-72](src/qbxml/parser.ts#L28-L72) now has `BillPaymentCheckRet`, `BillPaymentCreditCardRet`, `AppliedToTxnRet`, `PaymentRet`, `ReceivePaymentRet`. The full bill-payment shape round-trips through the parser correctly.
- **Verification-script gotcha** (round 2, surfaced this session): `vbal()` / `bal()` helpers initially used `ListFilter: { FullName: ... }` which the simulation does NOT recognize. The handler silently returns ALL records in the store, and `r[0]` is whichever record is first in iteration order — wrong for any non-first vendor. Fix: query all + `.find(v => v.FullName === name)`. The same gotcha applies for any future verification script that needs to look up a specific Customer/Vendor by name. Note that `nameFilter` (with `.NameFilter` wrapping) IS honored on the tool layer but the manager-level filter is positional. Use the all-then-find approach in verification scripts.
- **Real QB `BillPaymentCheck.PayeeEntityRef` vs simulation `VendorRef`.** Real QB's request shape uses `PayeeEntityRef` (which always points to a vendor for bill payments). The simulation uses `VendorRef` for compatibility with the existing `EntityFilter` handler in `handleQuery` (which checks `e.CustomerRef ?? e.VendorRef`). The tool layer takes `vendorName/vendorListId` from the operator, so the operator-facing API is unaffected. If live mode is ever wired, the builder will need to translate `VendorRef` → `PayeeEntityRef` for the BillPayment* request types specifically, or extend `handleQuery` to also check `PayeeEntityRef`. Document the choice in `DECISIONS.md` if/when this comes up.
- **No `qb_bill_payment_apply` analog.** Item 9 is Add-only. Re-targeting a bill payment to a different bill set (the AP analog of Item 8) is not currently in the todo list. If it's needed, the implementation would mirror Item 8's `handleReceivePaymentMod` exactly: validate-first, reverse-then-apply, bump EditSequence, atomic on failure. Probably a 100-line addition. Flag it in the next handoff if any operator workflow needs it.
