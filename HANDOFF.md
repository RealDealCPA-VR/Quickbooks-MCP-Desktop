- **`qb_journal_entry_duplicate`** ([src/tools/journal-entries.ts](src/tools/journal-entries.ts)) — GL-only analog. Per-line `Memo` / `EntityRef` / `ClassRef` carry through. Carries `IsAdjustment` flag (operator can override via `isAdjustment` arg). **No entity-retarget arg at the tool level** — re-pointing per-line EntityRef (Customer/Vendor on a JE line) is `qb_journal_entry_update` territory; a JE duplicate wholesale-carries the source's entity refs and the operator runs an Update after if they need to swap. **Design choice on line mapping:** I copy `AccountRef` / `Amount` / `Memo` / `EntityRef` / `ClassRef` field-for-field from the Ret rather than rebuilding via the tool's `buildJELineAdd` helper. The helper takes a flattened arg shape (`accountName: string` → `AccountRef: { FullName }`) and would lose `AccountRef.ListID` if the source carried only a ListID. The direct copy preserves whichever ref shape the source had.

## Context Notes

- **All three new `_duplicate` tools are pure composites** over `session.queryEntity` + `session.addEntity` / `session.addEntityIdempotent`. No new wire types, no manager changes, no simulation-store changes, no parser changes. Follows the same template as `qb_invoice_duplicate`. Any future `_duplicate` tool (CreditMemo, Estimate, PO, Check, Deposit, Transfer) plugs in as ~150 LOC of tool-handler scaffold + ~12 tests of the same 3-layer shape (happy / error / idempotency). The "field-for-field copy from Ret to Add" approach preserves `AccountRef.ListID` / `ItemRef.ListID` etc. when the source carries only a ListID rather than a FullName — important for live-mode duplicates where most refs come back ListID-only.
- **JE duplicate's "carry both sides verbatim" approach is what preserves the sum-balance invariant.** If you ever modify `qb_journal_entry_duplicate` to allow line-level overrides, the pre-persist balance check in handleAdd (statusCode 3030) will catch any inadvertent imbalance — but the natural composition through `addEntity("JournalEntry", …)` already validates it, so no new safety layer is needed.
- **`qb_sales_receipt_duplicate`'s `paymentMethodName` / `depositToAccountName` overrides are SR-specific** — the invoice/bill variants don't have analogous header-ref overrides because invoices and bills don't have header-level payment-method or deposit-account refs. If the operator wants to duplicate an invoice and retarget at a different sales rep, that's a future arg addition; for now they'd duplicate then update.
- **38 new tests use the "fake server / handler map" pattern** established by [tests/invoice-duplicate.test.ts](tests/invoice-duplicate.test.ts) — `registerXTools(fakeServer, () => session)` populates a `Map<string, Handler>`; each test pulls the handler by name and invokes it directly. No MCP transport in the loop. Fast (~1.3s for 634 tests). The pattern composes cleanly with `session.setReadOnly(true)` for 9001 gate testing — set the flag after seeding the source, then invoke the duplicate handler.
- **The bill test seed used "Office Supplies Co" instead of "Office Landlord LLC"** because the vendor-balance composition test queries the vendor BEFORE the bill is created (to capture the before-after delta), and `session.queryEntity("Vendor", {FullName})` returns empty for vendors that don't exist in the simulation seed at [simulation-store.ts](src/session/simulation-store.ts). This is a test-isolation choice — production tool code is fine for unseeded vendors (`adjustEntityBalance` silently no-ops when the vendor doesn't exist; the bill posts successfully without bumping any vendor balance). If you write tests for future `_duplicate` tools that need before-after entity-balance assertions, seed with a name from the existing seed: customers — "Acme Corporation" / "Global Industries"; vendors — "Office Supplies Co" / "Tech Solutions" / "Joe Contractor" / "Sarah Designer LLC" / "ACME Property Mgmt".
