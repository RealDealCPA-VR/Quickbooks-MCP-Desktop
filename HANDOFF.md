## Next Task

**Phase 4, Item 12 — CreditMemo family** (2 of 4 in Item 12). [todo.md:35](todo.md#L35).

The CreditMemo is the AR negative — credits the customer's open balance. Likely 5 tools (list/create/update/delete plus an apply path), tool count 57 → ~62.

### Recommended approach

1. **Standard CRUD shape first** — `qb_credit_memo_list` / `_create` / `_update` / `_delete`. Same line shape as Invoice (`CreditMemoLineAdd` with itemName/itemListId + quantity/rate). Same wholesale-replace `lines` semantics on update via `CreditMemoLineMod`. The `applyLineMods` regex catches it for free.

2. **`computeTotals` branch for CreditMemo** — derive `Subtotal = lineSum`, `TotalAmount = Subtotal + SalesTaxTotal`. Unlike SalesReceipt, CreditMemo DOES post to AR — but as a NEGATIVE balance (credit). Real QB tracks `RemainingValue` (= initial credit minus what's been applied to invoices). When fully applied, it disappears from "open credits." On a fresh credit memo (no application yet), `RemainingValue = TotalAmount`.

3. **Customer-balance bookkeeping** — `handleAdd` needs a CreditMemo branch in [simulation-store.ts:333](src/session/simulation-store.ts#L333). On add, customer Balance moves by `-TotalAmount` (the credit reduces what they owe). On `handleTxnDel` ([simulation-store.ts:1274](src/session/simulation-store.ts#L1274)), reverse with `+TotalAmount`. Use the `adjustPartyBalanceForTxn` helper but pass a NEGATIVE-direction sign — easiest is to either (a) extend the helper to accept a custom sign multiplier, or (b) post directly via `adjustEntityBalance` with `-totalAmount` delta. Option (b) is the cleaner break — the existing `adjustPartyBalanceForTxn` is hardcoded to `BalanceRemaining` / `AmountDue` field reads, but CreditMemo needs `TotalAmount`, so the helper doesn't fit cleanly anyway.

4. **`qb_credit_memo_apply`** — the AR-negative analog of `qb_payment_apply`. Re-apply an existing credit memo to a different set of open invoices via `CreditMemoMod` + `AppliedToTxnMod`. Same shape as `qb_payment_apply` ([src/tools/payments.ts:130-200](src/tools/payments.ts#L130-L200)) — pass `txnId` + `editSequence` plus a replacement `applyTo: [{txnId, amount}]` array. The new array REPLACES the prior application: previously-credited invoices get their BalanceRemaining restored, the new invoices receive the new credit. Customer balance moves by the change in applied sum (new applied − old applied), in the credit direction.

   **Important quirk** — CreditMemo's "AppliedToTxn" is conceptually the same shape as ReceivePayment's, but real QB uses a slightly different flow: the credit memo CAN auto-apply on creation (via `IsToBeAuto: true` in the AddRq) or remain as a customer credit pool. For the first cut, mirror `qb_payment_apply` exactly — `_apply` only re-targets existing credit memos, doesn't auto-apply on create. Auto-apply on create can be a follow-up.

5. **Simulation-store glue** — the existing `applyReceivePayment` at [simulation-store.ts:303-313](src/session/simulation-store.ts#L303-L313) is the closest precedent for an AR application that touches multiple invoices atomically. CreditMemo will need a similar `applyCreditMemo` helper in `handleAdd` (only fires when `AppliedToTxnAdd` is present in the create payload — most cases will be standalone credit memo creation). And a `handleCreditMemoMod` similar to `handleReceivePaymentMod` at [simulation-store.ts:973+](src/session/simulation-store.ts#L973) for the apply path. Validate-first then mutate (atomicity — orphan TxnID in line N must NOT corrupt the partial state).

6. **`qb_credit_memo_apply` validation** — sum of applied credits must not exceed the credit memo's TotalAmount. Mirror the `sumApplied > totalAmount` guard from `qb_payment_receive` ([src/tools/payments.ts:57-69](src/tools/payments.ts#L57-L69)).
