## Last Session Summary

- **Phase 12 #58 shipped — two new batch_create tools.** Generalizes the multi-request envelope plumbing #43 introduced for `qb_journal_entry_batch_create` to the invoice and sales-receipt surfaces. Replaces the Excel intermediate operators have historically used for bulk-load entry (monthly retainer billing, recurring subscription invoicing, POS day-end reconciliation, fundraiser bulk entry). Both tools are pure composites over the existing `session.executeBatchAdd` / `executeBatchAddIdempotent` primitives — no new manager methods, no parser changes, no sim-store changes, no new wire request types. The atomicity contract matches JE batch exactly: 1–100 entries per envelope, `onError="stopOnError"` halts on the first wire-side failure, and the tool's post-processing tears down any prior-posted entries in REVERSE post order via `session.deleteEntity(entityType, txnId)`.
- **`qb_invoice_batch_create`** ([src/tools/invoices.ts](src/tools/invoices.ts)) — AR-posting batch. Each entry takes the same shape as `qb_invoice_create` (customerName | customerListId, optional txnDate/dueDate/refNumber/memo, optional `lines` array). Upfront validation: every entry needs a customer ref; missing one rejects the WHOLE batch with statusCode 3120 BEFORE any wire I/O. On rollback, the underlying `handleTxnDel("Invoice", txnId)` reverses Customer.Balance by the deleted invoice's BalanceRemaining — so partial-failure paths leave AR consistent. Per-entry status enum: `posted` (committed), `rolled-back` (was posted, auto-deleted), `orphaned` (was posted but the rollback delete itself failed — operator cleans up via `qb_invoice_delete`), `failed` (wire-side rejection), `skipped` (post-stopOnError). On full success the response carries `success: true + count + invoices: [...]`; on partial failure `success: false + atomic: true + rolledBack: bool + failedAt + failedReason + summary + invoices: [...]` (with `isError: true`).
- **`qb_sales_receipt_batch_create`** ([src/tools/sales-receipts.ts](src/tools/sales-receipts.ts)) — cash-sale batch. Same shape as the invoice batch tool, accepts `paymentMethodName` / `depositToAccountName` / `depositToAccountListId` per entry (matches single-entry `qb_sales_receipt_create`). Rollback is structurally simpler — `handleTxnDel("SalesReceipt", txnId)` is a pure record removal (cash sales never posted to AR, so there's no Customer.Balance reversal to chase). All other tool-side machinery (upfront validation, stopOnError handling, REVERSE-order rollback dispatch, status enum, idempotency) is identical to the invoice variant.
- **Idempotency at the BATCH level (not per-entry).** Both tools accept an optional `idempotencyKey: string` that fingerprints the entire entries list — replay with the same key + identical list returns `idempotentReplay: true` and the cached batch outcome verbatim (cache lives in `QBSessionManager.idempotencyCache` keyed by `Batch:Invoice` / `Batch:SalesReceipt`). Reordering, adding, or removing entries changes the fingerprint and surfaces statusCode 9002 — use a fresh key. Only fully-successful batches are cached (carried over from `executeBatchAddIdempotent`'s contract — partial-failure outcomes aren't cached because replaying the pre-rollback wire outcome would cause the rollback path to re-attempt deletes of already-removed TxnIDs). The upfront customer-ref validation gate runs BEFORE the idempotency check; entries rejected upfront are never cached regardless of key, so the operator can retry the same key with a corrected payload without a 9002 conflict.
- **No schema-order risk to verify on Windows.** Both new tools reuse the InvoiceAdd / SalesReceiptAdd request bodies that are already pinned by existing tests/builder-emit-order.test.ts emit-order assertions. The multi-request envelope structure (`<QBXMLMsgsRq onError="stopOnError"><InvoiceAddRq requestID="1">…</InvoiceAddRq><InvoiceAddRq requestID="2">…</InvoiceAddRq>…</QBXMLMsgsRq>`) is the same shape JE batch already validated live. ## Context Notes

- **`qb_invoice_batch_create` and `qb_sales_receipt_batch_create` are pure composites** over `session.executeBatchAdd` / `executeBatchAddIdempotent` + `session.deleteEntity` (for compensating delete). No new wire request types. The infrastructure was generalized by #43 — `executeBatchAdd(entityType, entries)` is fully entity-type-generic; the simulation store's `processRequest` walks array-valued request keys via the same `outer: for (const [key, value] of Object.entries(msgsRq))` loop regardless of entity type. **Any future `_batch_create` tool** (Bill, CreditMemo, Estimate, PO, ReceivePayment, Check, Deposit) plugs in as ~250 LOC of tool-handler scaffold + ~12 tests of the same 4-layer shape (full success / upfront validation / partial-failure rollback / idempotency). Same envelope, same compensating-delete pattern.
- **Customer.Balance composition is the only structural difference between invoice and SR batches.** Invoice batches: each posted entry's `handleAdd("Invoice")` increments `Customer.Balance` by `BalanceRemaining`; on rollback `handleTxnDel("Invoice")` decrements it back. SR batches: cash sales never moved Customer.Balance in the first place (the deposit posts to the named DepositToAccountRef, not AR), so rollback is a pure record removal. The invoice-batch test pin asserts the AR composition by walking the customer's balance before/after a 3-entry batch (1500 increase); the SR-batch test asserts the inverse (no AR move).
- **Partial-failure rollback testing uses `vi.spyOn(session, "executeBatchAdd").mockResolvedValueOnce(...)`** to inject a tailored wire-side result (posted/failed/skipped mix) without needing a sim-side rejection path that natural inputs can reach. This is cleaner than the JE batch test's approach (which uses a zero-amount JE as a real wire-side trigger via `validateJournalEntryBalance`) because invoice/SR don't have an analogous natural-rejection path reachable from valid input. The "orphaned" test follows the same pattern but additionally mocks `session.deleteEntity` to throw, simulating QB rejecting the TxnDelRq (e.g., the invoice already had a payment applied to it before rollback fired). Both tests verify the deleteEntity call ORDER explicitly via `deleteSpy.mock.calls` (reverse post order).
- **Idempotency: `Batch:Invoice` / `Batch:SalesReceipt` cache prefixes** keep the batch fingerprint namespace separate from single-entry `Invoice` / `SalesReceipt` keys. The same idempotencyKey can theoretically be reused across a single-entry `qb_invoice_create` and a `qb_invoice_batch_create` call because their cached entityTypes differ — but this is incidental, not a contract. The validation-not-cached behavior is critical: an operator with a key like `"march-2026-retainers"` can call the batch tool with a typo'd entries list, get a 3120 rejection, fix the typo, and retry with the same key — the corrected payload succeeds because the rejected one was never cached. The fingerprint is order-sensitive: reordering entries within the same list changes the hash and surfaces 9002 (carried over from `executeBatchAddIdempotent` — a reordered batch is a different operation, NOT a replay).
