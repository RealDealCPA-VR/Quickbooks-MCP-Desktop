## Context Notes

- **What `qb_bill_create` looks like today** ([src/tools/bills.ts:52-100](src/tools/bills.ts#L52-L100)): accepts vendor ref, dates, refNumber, memo, and a flat `amountDue` number. There is no way to pass line items today, which means bills don't actually post to GL accounts. The acceptance criterion requires both `expenseLines` (each with `accountName` + `amount` + optional `memo` / `classRef`) and `itemLines` (each with `itemName` + `quantity` + `cost` + optional `memo`), and at least one of them must be present (header-only bills are rejected with a clear error).
- **Shape to build.** Real QBXML's `BillAdd` carries `ExpenseLineAdd` and `ItemLineAdd` blocks as siblings. Each `ExpenseLineAdd` needs `AccountRef` (FullName or ListID) + `Amount`. Each `ItemLineAdd` needs `ItemRef` + `Quantity` + `Cost`. The simulation already knows how to convert `*LineAdd` arrays to `*LineRet` ([src/session/simulation-store.ts:312-372](src/session/simulation-store.ts#L312-L372)) — the parser already registers `ExpenseLineRet` and `ItemLineRet` in `arrayElements` per Item 17 — so the tool just needs to build the right JS shape and pass it to `session.addEntity("Bill", data)`. No simulation-store changes required.
- **`Amount` rule for `ItemLineAdd`.** Per Item 17's implementation note at [src/session/simulation-store.ts:374-393](src/session/simulation-store.ts#L374-L393), `ItemLineAdd` lines that supply only `Quantity + Cost` (without explicit `Amount`) currently get `Amount = 0`. The line-converter only computes `Quantity * Rate`, not `Quantity * Cost`. Item 4's tool needs to either: (a) compute `Amount = quantity * cost` in the tool layer before passing to `session.addEntity`, or (b) extend the line-converter to also do `Quantity * Cost` when no `Rate` is present. **(a) is simpler and more local** — bill is the only entity using Cost-based item lines today, and pushing the math into the tool keeps the converter honest about real QB's behavior (real QB does compute `Amount` server-side, but only when the request omits it AND the math is unambiguous). Either approach satisfies the acceptance criterion that `AmountDue` equals sum of all line amounts.
- **Vendor balance update.** Item 18's helper `adjustPartyBalanceForTxn` at [src/session/simulation-store.ts:489-508](src/session/simulation-store.ts#L489-L508) already handles `Bill` → `Vendor` / `AmountDue` on add. Once Item 4 lands, vendor balances will move on bill creation with non-zero line totals — verify that `qb_vendor_list` reflects it.
- **AP aging.** Item 16's `computeTotals` already sets `Bill.AmountDue = sum(line amounts)` if not explicitly provided ([src/session/simulation-store.ts:425-427](src/session/simulation-store.ts#L425-L427)). Once line items flow through, `qb_ap_aging` should reflect non-zero open AP. (`qb_ap_aging`'s `asOfDate` is still decorative per Phase 5 item 19 — that's separate.)
- **Header-only rejection.** The acceptance criterion "at least one of `expenseLines` or `itemLines` is required" must be enforced in the tool layer with a clean `isError: true` response — the simulation will happily store a no-line bill (it'll just have `AmountDue = 0`), so don't rely on the simulation to reject. Pattern to follow: the existing `vendorName/vendorListId` check at [src/tools/bills.ts:67-78](src/tools/bills.ts#L67-L78).
- **`amountDue` arg interaction.** The current `qb_bill_create` already accepts an explicit `amountDue` arg (passed through to `data.AmountDue`). Item 16's `computeTotals` honors an explicit `AmountDue` and only computes when undefined ([src/session/simulation-store.ts:425-427](src/session/simulation-store.ts#L425-L427)). Decision call: keep `amountDue` as an optional override (operator can lock the total), OR remove it entirely now that lines drive it. **Recommended: remove it** — having both a derived total and an override total opens "which wins?" confusion and the acceptance criterion explicitly says `AmountDue = sum(lines)`. If you keep it, document the precedence in a comment. If you remove it, mention the schema breaking change in `DECISIONS.md`.
- **No new dependencies.** Per CLAUDE.md, any new dep needs a `DECISIONS.md` entry. Item 4 doesn't need any.
- **Project conventions reminder** ([CLAUDE.md](CLAUDE.md) § Stable Code Conventions): TS strict, ESM with `.js` extensions on relative imports, every zod field has `.describe()`, no comments explaining WHAT, no backwards-compat shims in tools.
