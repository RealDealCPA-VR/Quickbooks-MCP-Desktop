First write failure (disk full, permission, etc.) latches `disabled=true` and stderr-warns once; all subsequent calls no-op. Never poisons request flow. Directory creation lazy + once-per-rollover; mkdir failure also latches disabled.

## Context Notes

- **The logger is the diagnostic primitive for every future schema-order class of bug.** When a live tool surfaces `statusCode -1 — QuickBooks found an error when parsing the provided XML text stream`, the diagnostic flow is now: (1) check that `QB_DEBUG_QBXML=1` was set (or set it and reproduce); (2) read the last `op=request` block in the log; (3) compare against the canonical schema-order pinned in [tests/builder-emit-order.test.ts](tests/builder-emit-order.test.ts) for the offending request type; (4) the field that's out of order or missing is the bug. This replaces the prior workflow (add `console.error` to `manager.sendRequest`, rebuild, reproduce, remove the log, rebuild again) with a zero-rebuild path. The same flow surfaces the actual section / row labels for any future report-adapter mismatch (P&L, BS, SCF — and the parallel work for future reports like `qb_w2_summary` / payroll summary / sales tax liability that emit different row taxonomies).
- **Logger never throws into request flow.** A write failure (full disk, permission, etc.) latches `disabled=true` and warns once to stderr; subsequent calls no-op. So the worst case of "logger broken" is "no logs," never "tool calls start failing." If the operator reports no logs are being written despite `QB_DEBUG_QBXML=1`, the stderr line `[QbxmlLogger] Write to <path> failed (<reason>); disabling logger for the rest of the process.` will be in the MCP server's process output (Claude Desktop's `logs/` folder).
- **Redaction list is conservative (4 fields).** Adding a new tag: append to `SENSITIVE_TAGS` in [src/util/qbxml-logger.ts](src/util/qbxml-logger.ts) and add a corresponding assertion in the redaction test in [tests/qbxml-logger.test.ts](tests/qbxml-logger.test.ts). Don't over-extend — `Phone` / `Email` / `Address1` / `Memo` are personally-identifying but not credentials; redacting them would obscure legitimate debugging signal without meaningfully improving safety.
- **Sim mode response logging is JSON-stringified, not XML.** The `SimulationStore.processRequest` returns a parsed `QBXMLResponse` directly. If a future bug requires byte-for-byte comparison between live and sim wire shapes, the sim path would need a serializer to re-emit the response as XML — out of scope today; the parsed JSON shape is what every caller consumes anyway.
- **`resetQbxmlLogger()` is test-only.** Production code should never call it — there's no use case for swapping the logger mid-process. Tests use it via `beforeEach` / `afterEach` to handle env-var changes.
