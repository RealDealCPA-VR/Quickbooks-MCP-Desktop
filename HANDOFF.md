- **Stripe-style fingerprint matching.** Same key + matching payload fingerprint ‚Üí return cached result with `replayed: true`. Same key + different payload ‚Üí throw `QBIdempotencyKeyConflictError`. The fingerprint is SHA-256 of canonicalized JSON: recursive sorted-key normalization so `{Name, Phone}` and `{Phone, Name}` collide intentionally (they ARE the same request), but array order IS preserved (line[0] vs line[1] map to different TxnLineIDs after persist).
- **Failed creates do NOT poison the cache.** Operator's mental model: idempotency protects against *successful* duplicates, not against retrying *failed* writes. If the first call throws (validation, network, QB rejection), the next retry can succeed under the same key.
- **Tool surface ‚Äî 16 `*_create` / `*_add` handlers across 14 tool files** gain optional `idempotencyKey: z.string().min(1).optional()`. Coverage: customer/vendor/account/employee/item add; invoice/bill/SR/CM/estimate/PO/JE create; payment_receive; bill_pay; estimate_convert_to_invoice (the internal Invoice add); journal_entry_batch_create. On replay the response carries `idempotentReplay: true`.
- **Batch idempotency only caches full-success outcomes ‚Äî by design.** A partial-failure batch runs compensating-delete at the tool layer AFTER `executeBatchAdd` returns. Caching the pre-rollback wire outcome and replaying it would re-attempt deletes against TxnIDs the original call already removed ‚Äî observable thrash and a divergent response shape from the first call. The simpler and correct semantic: cache only when every entry posted, otherwise let the retry build a fresh envelope. Documented in tool description on `qb_journal_entry_batch_create`. Upfront balance-validation rejections are also not cached (a corrected retry under the same key works).
- **`qb_estimate_convert_to_invoice` skips the IsAccepted flip on replay.** The convert tool is a two-step composite (Invoice add + Estimate IsAccepted modify); only the InvoiceAdd half is keyed. On replay, re-running the EstimateMod with the original `editSequence` would either be a no-op (if already accepted) or fail with `statusCode 3170` (stale editSequence after the original mark). Skipping is safe and matches the operator's mental model: "the conversion already happened, give me back the invoice."

## Context Notes

- **The idempotency strategy is structural, not opportunistic.** Mirrors the #42 read-only gate template exactly: policy concerns affecting every mutation belong in `QBSessionManager`, not duplicated across 47 tool call sites. Future mutation tools (Phase 17 banking primitives, sales orders, etc.) inherit idempotency for free if they route through `addEntityIdempotent` instead of the bare `addEntity`. Before adding a new mutation tool, decide whether it deserves an `idempotencyKey` arg ‚Äî for any tool that creates a record an agent might want to retry safely, the answer is yes.
- **`statusCode 9002` is the second synthetic sentinel** (after #42's 9001 for read-only). 9003+ is reserved for future client-side gates. Both are mapped in [src/util/qb-status-codes.ts](src/util/qb-status-codes.ts) and surfaced through the existing tool error wrapper.
- **Per-companyFile scoping is structurally important.** A TxnID issued under company A is meaningless under company B even with the same idempotency key. `switchCompanyFile` clears the cache deliberately. If a future tool ever bypasses `switchCompanyFile` to mutate `config.companyFile` directly (which would be a bug ‚Äî `config` is private), the idempotency cache would silently leak across companies. The gate is structurally load-bearing the same way the read-only gate is.
- **Cache is in-memory and ephemeral.** Restarting the MCP server clears it. An agent that retries across a server crash gets a duplicate. Acceptable for personal use; a shared multi-process deployment would need persistent storage (Redis, SQLite). Not required.
- **The 1000-entry FIFO bound is sized for realistic CPA workflows.** Even January 1099 prep posts at most a few hundred bills per session. If a future workflow ever pushes past 1000 keyed creates per company-file session, older keys age out silently ‚Äî surface this in the tool description if real workloads grow.
- **`qb_estimate_convert_to_invoice` is a special case.** The convert tool is a composite: Invoice add + Estimate IsAccepted modify. Only the InvoiceAdd half is idempotency-keyed.
